# AULA 3.2: Pedindo Revisao (Pull Request)

**Modulo:** 3 - Colaborando com Sua Equipe
**Duracao:** 15 minutos
**Tipo:** Pratico

---

## GPS DA AULA

### DESTINO (Onde voce vai chegar)
Ao final desta aula, voce vai:
- Entender o que e Pull Request
- Saber propor mudancas para serem integradas
- Criar Pull Requests com descricoes claras
- Fazer a integracao (merge) das mudancas

### ORIGEM (De onde voce vem)
Voce provavelmente:
- Sabe criar branches (aula anterior)
- Nunca usou Pull Request
- Quer uma forma organizada de integrar mudancas

### ROTA (Como vamos chegar)
1. Entender o que e Pull Request
2. Criar um Pull Request
3. Escrever uma boa descricao
4. Fazer a integracao (merge)

## ROTEIRO COMPLETO

### [ABERTURA] - 30 segundos

**[FALA]**

> Voce trabalhou em uma branch separada. As mudancas estao prontas.
>
> E agora? Como integrar essas mudancas na versao principal?
>
> Para isso existe o Pull Request. E uma proposta formal: "aqui estao minhas mudancas, vamos revisar e integrar".
>
> Nesta aula, voce vai aprender a usar essa ferramenta.

---

### [O QUE E PULL REQUEST] - 1 minuto 30 segundos

[SLIDE 1 - Fluxo do PR]

**[FALA]**

> Pull Request, ou PR, e uma proposta de mudanca.
>
> O nome vem do ingles: voce esta "pedindo" (request) para "puxar" (pull) suas mudancas para a versao principal.
>
> O fluxo funciona assim:
>
> 1. Voce trabalha em uma branch separada
> 2. Quando termina, abre um Pull Request
> 3. Voce (ou outra pessoa) revisa as mudancas
> 4. Se estiver tudo certo, integra na versao principal
>
> [SLIDE 2 - Por que usar]
>
> Por que nao integrar direto?
>
> Porque o Pull Request permite revisar antes. Voce ve exatamente o que vai mudar antes de integrar.
>
> Isso e especialmente util quando:
>
> - Voce contratou um freelancer e quer ver o trabalho antes de aceitar
> - Voce mesmo quer revisar suas mudancas com calma
> - Mais de uma pessoa trabalha no projeto
>
> [Pausa]
>
> O Pull Request e o lugar onde voce decide: "essas mudancas estao boas para integrar?"

---

### [CRIANDO UM PULL REQUEST] - 2 minutos

[SLIDE 3 - GitHub interface]

**[FALA]**

> Depois que voce envia sua branch para o GitHub, criar o Pull Request e simples.
>
> Va ate o GitHub no navegador. Se voce fez push recentemente, vai aparecer um aviso amarelo: "sua-branch had recent pushes - Compare & pull request".
>
> Clique nesse botao.
>
> [SLIDE 4 - Formulario]
>
> Na tela de criacao, voce vai ver:
>
> **Base:** main - e onde voce quer integrar as mudancas
>
> **Compare:** sua branch - e de onde vem as mudancas
>
> **Titulo:** escreva o que essas mudancas fazem
>
> **Descricao:** explique com mais detalhes
>
> [Pausa]
>
> Se o aviso amarelo nao aparecer, va manualmente em: Pull requests → New pull request → Selecione as branches.
>
> Depois de preencher, clique em "Create pull request".

---

### [ESCREVENDO UMA BOA DESCRICAO] - 2 minutos

[SLIDE 5 - Estrutura da descricao]

**[FALA]**

> Uma boa descricao ajuda a entender as mudancas rapidamente.
>
> O formato sugerido:
>
> ```
> ## O que mudou
>
> Descreva em poucas frases o que voce fez.
>
> ## Por que mudou
>
> Explique o motivo da mudanca.
>
> ## O que foi alterado
>
> Liste os pontos principais:
> - Item 1
> - Item 2
> - Item 3
> ```
>
> [SLIDE 6 - Exemplo]
>
> Exemplo pratico:
>
> ```
> ## O que mudou
>
> Reorganizei o manual de processos de vendas.
>
> ## Por que mudou
>
> O manual antigo estava confuso e as etapas
> nao estavam na ordem correta.
>
> ## O que foi alterado
>
> - Reordenei as secoes por ordem cronologica
> - Adicionei secao de perguntas frequentes
> - Corrigi informacoes de contato desatualizadas
> ```
>
> [Pausa]
>
> Com essa descricao, qualquer pessoa entende o que foi feito e por que.

---

### [REVISANDO AS MUDANCAS] - 1 minuto 30 segundos

[SLIDE 7 - Abas do PR]

**[FALA]**

> Depois de criar o Pull Request, voce pode revisar as mudancas.
>
> O PR tem algumas abas uteis:
>
> **Conversation:** discussao geral sobre as mudancas
>
> **Commits:** lista de versoes que estao sendo propostas
>
> **Files changed:** mostra exatamente o que mudou em cada arquivo
>
> [Pausa]
>
> Na aba "Files changed", voce ve:
>
> - Linhas em verde: o que foi adicionado
> - Linhas em vermelho: o que foi removido
>
> Isso ajuda a revisar todas as mudancas antes de integrar.
>
> [SLIDE 8 - Fazendo ajustes]
>
> Se voce perceber que precisa mudar algo, nao precisa criar outro PR.
>
> Basta fazer as mudancas na mesma branch, salvar uma nova versao e enviar para o GitHub:
>
> ```
> git add .
> git commit -m "Corrige detalhes apontados na revisao"
> git push
> ```
>
> O Pull Request atualiza automaticamente com as novas mudancas.

---

### [INTEGRANDO AS MUDANCAS] - 2 minutos

[SLIDE 9 - Botao Merge]

**[FALA]**

> Quando as mudancas estao prontas para integrar, voce faz o "merge".
>
> No Pull Request, procure o botao verde "Merge pull request".
>
> Clique nele e depois em "Confirm merge".
>
> [Pausa]
>
> Pronto. Suas mudancas agora fazem parte da versao principal.
>
> [SLIDE 10 - Limpeza]
>
> Depois do merge, o GitHub oferece a opcao "Delete branch".
>
> Clique para apagar a branch que voce nao precisa mais. Isso mantem o projeto organizado.
>
> No seu computador, atualize a versao principal:
>
> ```
> git checkout main
> git pull
> ```
>
> E apague a branch local:
>
> ```
> git branch -d nome-da-branch
> ```
>
> Agora tudo esta sincronizado.

---

### [PRATICA] - 3 minutos

[SLIDE 11 - Pratica]

**[FALA]**

> Vamos criar um Pull Request de verdade.
>
> Se voce fez a aula anterior, ja tem uma branch publicada no GitHub.
>
> Va ate o GitHub no navegador. Voce deve ver o aviso amarelo. Clique em "Compare & pull request".
>
> [Pausa]
>
> Preencha o formulario:
>
> **Titulo:** Adiciona nova secao ao README
>
> **Descricao:**
>
> ```
> ## O que mudou
>
> Adicionei uma nova secao ao README para testar
> o fluxo de Pull Request.
>
> ## Por que mudou
>
> Pratica da aula 3.2 do curso GitHub Essencial.
>
> ## O que foi alterado
>
> - Nova secao adicionada ao final do README
> ```
>
> Clique em "Create pull request".
>
> [Pausa]
>
> Agora va ate a aba "Files changed" e veja suas mudancas.
>
> Quando estiver pronto, clique em "Merge pull request" e depois "Confirm merge".
>
> Clique em "Delete branch" para limpar.
>
> No terminal, atualize:
>
> ```
> git checkout main
> git pull
> ```
>
> Suas mudancas agora estao na versao principal.

### [FECHAMENTO] - 30 segundos

[SLIDE 12 - Resumo]

**[FALA]**

> Recapitulando:
>
> - Pull Request e uma proposta de mudanca
> - Permite revisar antes de integrar
> - Escreva descricao clara: o que, por que, o que mudou
> - Depois do merge, delete a branch
>
> Pull Request e a forma profissional de integrar mudancas.
>

## SE DER ERRO

### "There isn't anything to compare"

As branches sao identicas ou voce nao fez commits na branch.

Verifique se fez mudancas e salvou:

```
git log --oneline
```

### Branch nao aparece no GitHub

Voce esqueceu de enviar a branch:

```
git push -u origin nome-da-branch
```

### "Merge conflicts"

Ha conflitos entre as versoes. Veja a aula 3.4 para resolver.

## COMANDOS DA AULA (COPIAR E COLAR)

```bash
# Enviar branch para GitHub (se ainda nao fez)
git push -u origin nome-da-branch

# Fazer ajustes e atualizar o PR
git add .
git commit -m "Corrige detalhes"
git push

# Depois do merge, atualizar local
git checkout main
git pull

# Deletar branch local
git branch -d nome-da-branch
```
