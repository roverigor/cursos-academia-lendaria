### [O MODELO MENTAL] - 2 minutos

[SLIDE 1 - Diagrama das 3 áreas]

**[PROFESSOR DIZ:]**

> Git funciona com 3 áreas. Você precisa entender isso porque senão os comandos não fazem sentido.
>
> **Primeira área: Working Directory.** É onde você edita seus arquivos. Você abre o VS Code, faz suas mudanças, aqui que tá o rascunho.
>
> **Segunda área: Staging Area.** É onde você prepara o que vai ser salvo. Pensa assim: você tá montando um pacote. Você decide o que vai dentro.
>
> **Terceira área: Repository.** É o histórico salvo. Quando você faz commit, a foto fica registrada aqui. Permanente.
>
> [SLIDE 2 - Analogia]
>
> Analogia: Working Directory é seu rascunho. Staging Area é o envelope que você tá preparando. Repository é a carta enviada - não tem volta.
>
> O fluxo é sempre: editar → add → commit. Editar, add, commit. Você vai repetir isso centenas de vezes.

---

### [GIT STATUS] - 2 minutos

[SLIDE 3 - Terminal]

**[PROFESSOR DIZ:]**

> Primeiro comando: `git status`. Esse é seu GPS. Mostra onde você tá, o que mudou, o que tá preparado.
>
> Use o tempo todo. Antes de add, depois de add, antes de commit, quando tiver perdido.
>
> ```bash
> git status
> ```
>
> [SLIDE 4 - Saída do git status]
>
> A saída parece complicada, mas vou traduzir:
>
> "Changes not staged for commit" = arquivos modificados, mas não estão na staging area. Você precisa dar add.
>
> "Untracked files" = arquivos novos que o Git ainda não conhece. Também precisa de add.
>
> "Changes to be committed" = pronto pra commit. Tá na staging area.
>
> "nothing to commit, working tree clean" = tudo salvo. Você pode ir embora tranquilo.
>
> Versão curta:
>
> ```bash
> git status -s
> ```
>
> Mostra só duas letrinhas por arquivo. M = modificado. ?? = novo.

---

### [GIT ADD] - 2 minutos

[SLIDE 5 - git add]

**[PROFESSOR DIZ:]**

> Segundo comando: `git add`. Ele move arquivos do Working Directory pra Staging Area.
>
> Você pode adicionar arquivo específico:
>
> ```bash
> git add index.js
> ```
>
> Ou adicionar tudo de uma vez:
>
> ```bash
> git add .
> ```
>
> O ponto significa "tudo nessa pasta".
>
> [Pausa]
>
> Dica: na maioria das vezes, usa `git add .` mesmo. É mais prático.
>
> Depois do add, dá um `git status` pra confirmar. Você deve ver "Changes to be committed" com seus arquivos.
>
> Se adicionou arquivo errado:
>
> ```bash
> git restore --staged arquivo.js
> ```
>
> Isso tira do staging sem perder as mudanças.

---

### [GIT COMMIT] - 3 minutos

[SLIDE 6 - git commit]

**[PROFESSOR DIZ:]**

> Terceiro comando: `git commit`. Esse salva a foto no histórico.
>
> O mais comum é assim:
>
> ```bash
> git commit -m "Add login feature"
> ```
>
> O `-m` significa message. Entre aspas vai sua mensagem de commit.
>
> [SLIDE 7 - Boas mensagens]
>
> Agora, presta atenção nas mensagens. Isso é mais importante do que parece.
>
> Use imperativo: "Add", "Fix", "Update". Não "Added", "Fixed", "Updated".
>
> Por quê? Porque commit descreve o que ele FAZ quando aplicado, não o que você FEZ.
>
> Seja específico:
>
> - Ruim: "changes"
> - Ruim: "update"
> - Ruim: "fix bug"
> - Bom: "Fix navbar alignment on mobile"
> - Bom: "Add user authentication with JWT"
>
> [Pausa]
>
> Recrutador que abre seu repo vê essas mensagens. Mensagens genéricas = amador. Mensagens específicas = profissional.

---

### [O CICLO COMPLETO] - 2 minutos

[SLIDE 8 - Fluxograma do ciclo]

**[PROFESSOR DIZ:]**

> Vamos juntar tudo. O ciclo que você vai repetir dezenas de vezes por dia:
>
> ```bash
> # 1. Veja situação atual
> git status
>
> # 2. Edite seus arquivos no editor
>
> # 3. Veja o que mudou
> git status
>
> # 4. Adicione pra staging
> git add .
>
> # 5. Confirme que tá certo
> git status
>
> # 6. Salve no histórico
> git commit -m "sua mensagem"
>
> # 7. Confirme que tá limpo
> git status
> ```
>
> Status, add, commit. Status, add, commit.
>
> E pra ver seu histórico:
>
> ```bash
> git log --oneline
> ```
>
> Mostra uma linha por commit. Hash + mensagem.

**[PROFESSOR DIZ:]**

>

> ```

>

> Veja o histórico:

## TROUBLESHOOTING

### "nothing to commit"

> Você esqueceu o `git add`. Roda:
> ```bash
> git add .
> git commit -m "mensagem"
> ```

### Commitou arquivo errado

> Antes de fazer push:
> ```bash
> git reset --soft HEAD~1
> ```
> Volta o commit mas mantém as mudanças.
