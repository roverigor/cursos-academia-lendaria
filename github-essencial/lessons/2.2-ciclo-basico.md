### [COMO O GIT ORGANIZA AS MUDANCAS] - 2 minutos

[SLIDE 1 - As 3 areas]

**[FALA]**

> O Git organiza seu trabalho em 3 areas. Vou explicar cada uma de forma simples.
>
> **Area 1: Pasta de trabalho**
>
> E a pasta do seu projeto no computador. E onde voce edita seus arquivos normalmente.
>
> Quando voce abre um documento e faz alteracoes, as mudancas estao aqui.
>
> **Area 2: Area de preparacao**
>
> E como uma "caixa" onde voce coloca os arquivos que quer salvar.
>
> Voce decide o que vai entrar na proxima versao.
>
> **Area 3: Historico**
>
> E onde as versoes ficam salvas permanentemente.
>
> Quando voce confirma uma versao, ela vai para o historico e fica la para sempre.
>
> [SLIDE 2 - Analogia]
>
> Pense assim:
>
> A pasta de trabalho e sua mesa com papeis espalhados.
>
> A area de preparacao e um envelope onde voce coloca o que quer guardar.
>
> O historico e o arquivo onde o envelope fica guardado.
>
> [Pausa]
>
> O processo e sempre: editar → preparar → salvar. Voce vai repetir isso varias vezes.

---

### [COMANDO 1: GIT STATUS] - 2 minutos

[SLIDE 3 - git status]

**[FALA]**

> O primeiro comando e `git status`. Ele mostra a situacao atual dos seus arquivos.
>
> Pense nele como um "raio-x" que mostra o que esta acontecendo.
>
> No terminal, dentro da pasta do seu projeto, digite:
>
> ```
> git status
> ```
>
> [SLIDE 4 - Resultado do status]
>
> O resultado pode parecer complicado no inicio, mas vou traduzir:
>
> **"Changes not staged for commit"** significa: arquivos foram modificados, mas ainda nao estao preparados para salvar. Voce precisa usar o comando "add".
>
> **"Untracked files"** significa: arquivos novos que o Git ainda nao conhece. Tambem precisa de "add".
>
> **"Changes to be committed"** significa: arquivos prontos para serem salvos.
>
> **"nothing to commit, working tree clean"** significa: tudo ja foi salvo. Nao tem mudancas pendentes.
>
> [Pausa]
>
> Use esse comando sempre que quiser saber em que pe esta seu projeto.

---

### [COMANDO 2: GIT ADD] - 2 minutos

[SLIDE 5 - git add]

**[FALA]**

> O segundo comando e `git add`. Ele prepara arquivos para serem salvos.
>
> Quando voce modifica ou cria um arquivo, ele fica na pasta de trabalho. O comando "add" move ele para a area de preparacao.
>
> Voce pode preparar um arquivo especifico:
>
> ```
> git add nome-do-arquivo.txt
> ```
>
> Ou preparar todos os arquivos de uma vez:
>
> ```
> git add .
> ```
>
> O ponto significa "todos os arquivos desta pasta".
>
> [Pausa]
>
> Na maioria das vezes, voce vai usar `git add .` para preparar tudo de uma vez. E mais pratico.
>
> Depois de usar o add, rode `git status` novamente para confirmar que os arquivos estao preparados.
>
> Se voce adicionou algo por engano, pode remover da area de preparacao com:
>
> ```
> git restore --staged nome-do-arquivo.txt
> ```
>
> Isso remove da preparacao, mas nao apaga suas mudancas.

---

### [COMANDO 3: GIT COMMIT] - 2 minutos

[SLIDE 6 - git commit]

**[FALA]**

> O terceiro comando e `git commit`. Ele salva a versao no historico.
>
> O formato mais comum e:
>
> ```
> git commit -m "sua mensagem aqui"
> ```
>
> O `-m` significa "message" (mensagem). Entre aspas, voce escreve uma descricao do que mudou.
>
> [SLIDE 7 - Boas mensagens]
>
> A mensagem e importante porque ela fica no historico. Quando voce olhar suas versoes antigas, vai querer saber o que cada uma faz.
>
> Exemplos de mensagens:
>
> - "Atualiza documento de processos"
> - "Adiciona nova secao de perguntas frequentes"
> - "Corrige erro no calculo de precos"
>
> Evite mensagens genericas como "alteracoes" ou "update". Seja especifico sobre o que mudou.
>
> [Pausa]
>
> Depois do commit, rode `git status` para confirmar. Deve aparecer "nothing to commit" - significa que tudo foi salvo.

---

### [O CICLO COMPLETO] - 2 minutos

[SLIDE 8 - O ciclo]

**[FALA]**

> Vamos juntar tudo. O ciclo que voce vai repetir sempre:
>
> ```
> # 1. Veja a situacao atual
> git status
>
> # 2. Edite seus arquivos normalmente
>
> # 3. Veja o que mudou
> git status
>
> # 4. Prepare os arquivos
> git add .
>
> # 5. Confirme que estao preparados
> git status
>
> # 6. Salve a versao
> git commit -m "sua mensagem"
>
> # 7. Confirme que foi salvo
> git status
> ```
>
> Parece muito, mas com o tempo fica automatico: status, add, commit.
>
> [Pausa]
>
> Para ver o historico de versoes que voce salvou:
>
> ```
> git log --oneline
> ```
>
> Esse comando mostra uma linha por versao, com a mensagem que voce escreveu.

---

### [PRATICA] - 3 minutos

[SLIDE 9 - Pratica]

**[FALA]**

> Vamos praticar juntos. Abra o terminal na pasta do seu repositorio.
>
> Primeiro, veja a situacao:
>
> ```
> git status
> ```
>
> Deve estar limpo, sem mudancas pendentes.
>
> Agora, abra o arquivo README.md no seu editor de texto. Adicione uma linha com qualquer texto. Salve o arquivo.
>
> Volte ao terminal e veja a situacao:
>
> ```
> git status
> ```
>
> Deve aparecer que o README.md foi modificado.
>
> Prepare o arquivo:
>
> ```
> git add .
> ```
>
> Veja novamente:
>
> ```
> git status
> ```
>
> Agora ele esta em "Changes to be committed".
>
> Salve a versao:
>
> ```
> git commit -m "Atualiza README com nova informacao"
> ```
>
> Confirme:
>
> ```
> git status
> ```
>
> Deve aparecer "nothing to commit". A versao foi salva.
>
> Veja o historico:
>
> ```
> git log --oneline
> ```
>
> Seu commit aparece na lista.

---

### [MAIS DUAS VERSOES] - 1 minuto

[SLIDE 10 - Mais pratica]

**[FALA]**

> Agora faca mais dois commits para praticar.
>
> Edite o README novamente. Adicione outra linha.
>
> ```
> git add .
> git commit -m "Adiciona segunda secao ao README"
> ```
>
> Faca uma terceira mudanca:
>
> ```
> git add .
> git commit -m "Melhora formatacao do README"
> ```
>
> Veja o historico:
>
> ```
> git log --oneline
> ```
>
> Voce deve ter 3 ou mais commits agora.
>
> Se conseguiu, parabens. Voce aprendeu o ciclo basico do Git.

## SE DER ERRO

### "nothing to commit"

Voce esqueceu de preparar os arquivos. Use:

```
git add .
git commit -m "mensagem"
```

### Salvou versao errada

Se ainda nao enviou para o GitHub, pode voltar atras:

```
git reset --soft HEAD~1
```

Isso desfaz o ultimo commit, mas mantem suas mudancas.
