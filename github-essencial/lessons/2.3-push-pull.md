### [O MODELO LOCAL VS REMOTO] - 1 minuto 30 segundos

[SLIDE 1 - Diagrama local vs remoto]

**[PROFESSOR DIZ:]**

> Presta atenção nesse diagrama porque ele explica tudo.
>
> Você tem duas cópias do repositório: uma no seu computador, outra no GitHub.
>
> O seu computador é o **local**. O GitHub é o **remoto**.
>
> Quando você faz commit, você salva no local. O GitHub não sabe que esse commit existe.
>
> [SLIDE 2 - Fluxo push/pull]
>
> Pra mandar pro GitHub, você usa **push**. Pensa: "empurrar" seus commits pra nuvem.
>
> Pra trazer do GitHub pro seu computador, você usa **pull**. Pensa: "puxar" as atualizações.
>
> [Pausa]
>
> Por que isso importa?
>
> Primeiro: backup. Se seu computador morrer, o código tá seguro no GitHub.
>
> Segundo: colaboração. Seu time só vê seu trabalho depois que você faz push.
>
> Terceiro: mobilidade. Você pode continuar trabalhando de outro computador.

---

### [GIT PUSH] - 2 minutos

[SLIDE 3 - git push]

**[PROFESSOR DIZ:]**

> Push envia seus commits locais pro GitHub.
>
> O comando mais simples:
>
> ```bash
> git push
> ```
>
> Se for a primeira vez que você tá enviando essa branch, usa assim:
>
> ```bash
> git push -u origin main
> ```
>
> O `-u` configura a conexão entre sua branch local e a remota. Depois disso, pode usar só `git push`.
>
> [SLIDE 4 - Saída do push]
>
> Quando funciona, você vê algo assim:
>
> ```
> Writing objects: 100% (3/3), done.
> To github.com:username/repo.git
>    abc1234..def5678  main -> main
> ```
>
> Esse `main -> main` significa: mandou da branch main local pra branch main do GitHub.
>
> [Pausa]
>
> Dica importante: nunca use `push --force` em branches compartilhadas. Isso pode sobrescrever o trabalho dos colegas. A gente vai falar mais sobre isso depois.

---

### [GIT PULL] - 2 minutos

[SLIDE 5 - git pull]

**[PROFESSOR DIZ:]**

> Pull faz o oposto. Baixa commits do GitHub pro seu computador.
>
> ```bash
> git pull
> ```
>
> Você usa pull quando alguém do time fez mudanças, ou quando você editou algo direto no GitHub.
>
> [SLIDE 6 - Pull com rebase]
>
> Eu recomendo configurar pull com rebase. O histórico fica mais limpo.
>
> ```bash
> git config --global pull.rebase true
> ```
>
> Agora todo pull vai usar rebase automaticamente.
>
> [Pausa]
>
> Qual a diferença?
>
> Pull normal cria um commit de merge. O histórico fica cheio de "Merge branch".
>
> Pull com rebase aplica seus commits em cima dos novos. Histórico linear, mais fácil de ler.
>
> Se isso parece confuso agora, não se preocupa. Configura o rebase e segue em frente. Você vai entender melhor com a prática.

---

### [QUANDO USAR CADA COMANDO] - 1 minuto

[SLIDE 7 - Tabela quando usar]

**[PROFESSOR DIZ:]**

> Regra simples:
>
> **Começo do dia:** pull. Atualiza seu código com o que o time fez.
>
> **Terminou uma feature:** push. Manda pro GitHub.
>
> **Antes de criar Pull Request:** pull. Garante que você tá atualizado.
>
> **Fim do dia:** push. Backup do seu trabalho.
>
> [Pausa]
>
> Se você seguir essa rotina, nunca vai ter problema de sincronização.
>
> E sempre que der erro no push, faz pull primeiro. 90% das vezes resolve.

### [EXERCÍCIO AO VIVO] - 2 minutos

[SLIDE 8 - Terminal ao vivo]

**[PROFESSOR DIZ:]**

> Vamos praticar. Abre o terminal no seu repo de prática.
>
> Primeiro, vamos criar uma mudança e fazer push:
>
> ```bash
> cd github-practice-repo
> echo "// Push practice" >> index.js
> git add .
> git commit -m "Add push practice comment"
> git push
> ```
>
> Confere no GitHub se o commit apareceu.
>
> [Pausa]
>
> Agora vamos simular uma mudança que veio de outro lugar.
>
> Vai no GitHub, clica no arquivo README.md, clica no lápis pra editar.
>
> Adiciona uma linha: "Editado via GitHub".
>
> Clica em "Commit changes".
>
> [Pausa]
>
> Agora volta pro terminal. Seu local tá desatualizado.
>
> ```bash
> git status
> ```
>
> Ainda não sabe da mudança. Pra saber, roda:
>
> ```bash
> git fetch
> git status
> ```
>
> Agora mostra que você tá atrás. Atualiza:
>
> ```bash
> git pull
> ```
>
> Pronto. Seu local tá sincronizado com o GitHub.

### [TROUBLESHOOTING] - 1 minuto

[SLIDE 9 - Erros comuns]

**[PROFESSOR DIZ:]**

> Dois erros que você vai encontrar.
>
> Primeiro: "rejected - non-fast-forward".
>
> Significa que o GitHub tem commits que você não tem. Solução:
>
> ```bash
> git pull --rebase
> git push
> ```
>
> Segundo: "Please commit your changes or stash them".
>
> Significa que você tem mudanças não commitadas e o pull iria sobrescrever. Solução:
>
> ```bash
> git stash
> git pull
> git stash pop
> ```
>
> Stash guarda suas mudanças temporariamente. A gente vai ver isso em detalhes no Módulo 4.

## TROUBLESHOOTING

### Erro: "rejected - non-fast-forward"

> GitHub tem commits que você não tem.
> ```bash
> git pull --rebase
> git push
> ```

### Erro: "Please commit your changes or stash them"

> Você tem mudanças não commitadas.
> ```bash
> git stash
> git pull
> git stash pop
> ```

### Erro: "Permission denied (publickey)"

> SSH não configurado.
> ```bash
> ssh -T git@github.com
> ```
> Se falhar, revise a aula 1.3.
