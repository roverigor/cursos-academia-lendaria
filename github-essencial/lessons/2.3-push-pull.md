**Modulo:** 2 - Seu Primeiro Projeto no GitHub
**Duracao:** 10 minutos

### DESTINO (Onde voce vai chegar)
Ao final desta aula, voce vai:
- Saber enviar seus arquivos para o GitHub
- Saber baixar atualizacoes do GitHub
- Entender o fluxo de sincronizacao
- Ter seus arquivos seguros na nuvem

### ORIGEM (De onde voce vem)
Voce provavelmente:

## ROTEIRO COMPLETO

### [ABERTURA] - 30 segundos

**[FALA]**

> Na aula anterior, voce aprendeu a salvar versoes dos seus arquivos.
>
> Mas essas versoes estao apenas no seu computador. Se algo acontecer com ele, voce perde tudo.
>
> Nesta aula, voce vai aprender a enviar essas versoes para o GitHub. Assim elas ficam seguras na nuvem.

---

### [COMPUTADOR VS NUVEM] - 1 minuto 30 segundos

[SLIDE 1 - Diagrama local vs nuvem]

**[FALA]**

> Vamos entender como funciona a sincronizacao.
>
> Voce tem duas copias do seu projeto: uma no seu computador, outra no GitHub.
>
> A copia no seu computador e chamada de **local**. A copia no GitHub e chamada de **remoto** (porque esta em outro lugar, na nuvem).
>
> [Pausa]
>
> Quando voce salva uma versao com o comando commit, ela fica apenas no seu computador. O GitHub ainda nao sabe que essa versao existe.
>
> [SLIDE 2 - Fluxo enviar/receber]
>
> Para mandar suas versoes para o GitHub, voce usa o comando **push**. A palavra vem do ingles e significa "empurrar". Voce esta "empurrando" seus arquivos para a nuvem.
>
> Para trazer atualizacoes do GitHub para o seu computador, voce usa o comando **pull**. Significa "puxar". Voce esta "puxando" as atualizacoes.
>
> [Pausa]
>
> Por que isso e importante?
>
> **Primeiro:** backup automatico. Se seu computador parar de funcionar, seus arquivos estao seguros no GitHub.
>
> **Segundo:** acesso de qualquer lugar. Voce pode trabalhar de outro computador quando precisar.
>
> **Terceiro:** colaboracao. Se outra pessoa trabalhar com voce, ela vai ver as atualizacoes que voce enviar.

---

### [ENVIANDO PARA O GITHUB] - 2 minutos

[SLIDE 3 - git push]

**[FALA]**

> Vamos comecar enviando suas versoes para o GitHub.
>
> O comando e simples:
>
> ```
> git push
> ```
>
> Se for a primeira vez que voce esta enviando, use assim:
>
> ```
> git push -u origin main
> ```
>
> [Pausa]
>
> Deixa eu explicar essa linha:
>
> - `git push` e o comando para enviar
> - `-u` configura a conexao (voce so precisa usar isso uma vez)
> - `origin` e o nome padrao do GitHub na configuracao
> - `main` e o nome da linha principal do seu projeto
>
> Depois da primeira vez, pode usar apenas `git push`.
>
> [SLIDE 4 - Resultado do push]
>
> Quando funciona, voce ve algo assim:
>
> ```
> Writing objects: 100% (3/3), done.
> To github.com:seu-usuario/seu-projeto.git
>    abc1234..def5678  main -> main
> ```
>
> Isso significa que suas versoes foram enviadas com sucesso.
>
> [Pausa]
>
> Depois de fazer push, va ate o GitHub no navegador e atualize a pagina. Voce vai ver suas mudancas la.

---

### [RECEBENDO DO GITHUB] - 2 minutos

[SLIDE 5 - git pull]

**[FALA]**

> O comando pull faz o caminho contrario. Ele baixa as atualizacoes do GitHub para o seu computador.
>
> ```
> git pull
> ```
>
> Quando voce usa esse comando?
>
> - Quando voce editou algo diretamente no site do GitHub
> - Quando outra pessoa fez mudancas no projeto
> - Quando voce quer garantir que esta com a versao mais recente
>
> [Pausa]
>
> [SLIDE 6 - Boas praticas]
>
> Uma boa pratica e fazer pull no comeco do dia de trabalho. Assim voce comeca sempre com a versao mais atualizada.
>
> Outra boa pratica e fazer push no final do dia. Assim suas mudancas ficam salvas na nuvem, mesmo que voce nao tenha terminado.
>
> [Pausa]
>
> Pense assim: pull no comeco, push no final. Esse e o habito que vai proteger seu trabalho.

---

### [PRATICA] - 2 minutos

[SLIDE 7 - Pratica]

**[FALA]**

> Vamos praticar. Abra o terminal na pasta do seu projeto.
>
> Primeiro, vamos criar uma mudanca e enviar para o GitHub:
>
> ```
> git status
> ```
>
> Se estiver limpo, vamos criar uma mudanca. Abra o arquivo README.md e adicione uma linha qualquer.
>
> Agora salve a versao:
>
> ```
> git add .
> git commit -m "Atualiza README com nova informacao"
> git push
> ```
>
> [Pausa]
>
> Va ate o GitHub no navegador e confira se a mudanca apareceu.
>
> [SLIDE 8 - Simulando atualizacao externa]
>
> Agora vamos simular uma atualizacao que veio de outro lugar.
>
> No GitHub, clique no arquivo README.md. Clique no icone de lapis para editar.
>
> Adicione uma linha: "Editado pelo GitHub".
>
> Role a pagina e clique em "Commit changes".
>
> [Pausa]
>
> Agora volte para o terminal. Seu computador ainda nao sabe dessa mudanca.
>
> ```
> git pull
> ```
>
> Pronto. Agora seu computador esta atualizado com o que esta no GitHub.
>
> Abra o arquivo README.md e confirme que a linha que voce adicionou no GitHub esta la.

---

### [SE DER PROBLEMA] - 1 minuto

[SLIDE 9 - Problemas comuns]

**[FALA]**

> Dois problemas que podem acontecer.
>
> **Primeiro:** voce tenta fazer push e aparece um erro dizendo que o GitHub tem mudancas que voce nao tem.
>
> A solucao e simples: faca pull primeiro, depois push.
>
> ```
> git pull
> git push
> ```
>
> **Segundo:** voce tenta fazer pull e aparece um erro dizendo que voce tem mudancas nao salvas.
>
> A solucao: salve suas mudancas primeiro com commit, depois faca pull.
>
> ```
> git add .
> git commit -m "Salva mudancas antes de atualizar"
> git pull
> ```
>
> [Pausa]
>
> Na maioria das vezes, a regra e: se der erro no push, faca pull primeiro. Resolve quase todos os problemas.

## SE DER ERRO

### Erro no push: "rejected" ou "non-fast-forward"

O GitHub tem atualizacoes que voce nao tem. Solucao:

```
git pull
git push
```

### Erro no pull: "Please commit your changes"

Voce tem mudancas nao salvas. Solucao:

```
git add .
git commit -m "Salva mudancas pendentes"
git pull
```

### Erro: "Permission denied (publickey)"

A conexao SSH nao esta funcionando. Revise a aula 1.3 sobre conexao segura.
