**Módulo:** 2 - Primeiro Repo Real
**Duração:** 10 minutos
**Tipo:** Prático

---

## GPS DA AULA

### DESTINO (Onde você vai chegar)
Ao final desta aula, você vai:
- Ter um repositório profissional no GitHub
- Entender o que são README, .gitignore e LICENSE
- Saber clonar o repo localmente via SSH

### ORIGEM (De onde você vem)
Você provavelmente:
- Tem Git instalado e SSH configurado (Módulo 1)
- Talvez já tenha criado repos mas sem estrutura certa
- Não sabe que README vazio afasta recrutadores

### ROTA (Como vamos chegar)
1. Criar repo no GitHub com estrutura profissional
2. Entender os 3 arquivos essenciais
3. Clonar localmente via SSH
4. Verificar que tudo funcionou

### [POR QUE ESTRUTURA IMPORTA] - 1 minuto

[SLIDE 1 - Comparativo de repos]

**[PROFESSOR DIZ:]**

> Recrutador abre seu GitHub. Ele tem 3 segundos pra decidir se vai clicar no seu repo ou ir pro próximo candidato.
>
> [SLIDE 2 - Repo ruim vs bom]
>
> Olha esse repo aqui. README vazio. Nenhuma descrição. Um monte de arquivos jogados. Você clicaria?
>
> Agora olha esse outro. README explicando o que é, como rodar, screenshots. .gitignore configurado. Licença clara.
>
> Qual passa mais confiança?
>
> 70% dos recrutadores de tech olham GitHub antes de chamar pra entrevista. A estrutura do seu repo é sua primeira impressão.

---

### [OS 3 ARQUIVOS ESSENCIAIS] - 2 minutos

[SLIDE 3 - Estrutura de repo profissional]

**[PROFESSOR DIZ:]**

> Todo repo profissional tem 3 arquivos na raiz. Vou mostrar cada um.
>
> **README.md** - O cartão de visitas do seu projeto. Explica o que é, pra que serve, como instalar, como usar. É a primeira coisa que as pessoas leem.
>
> **[SLIDE 4 - .gitignore]**
>
> **.gitignore** - Lista de arquivos que o Git deve ignorar. Dependências (tipo node_modules), arquivos de configuração local (.env), coisas do sistema.
>
> Por que isso importa? Porque sem .gitignore, você pode subir suas senhas e chaves de API pro GitHub. Já vi isso destruir carreiras.
>
> **[SLIDE 5 - LICENSE]**
>
> **LICENSE** - Define como outras pessoas podem usar seu código. Sem licença, ninguém pode usar legalmente, mesmo sendo público.
>
> Pra portfolio, usa MIT. É simples, permissiva, e empresas aceitam bem.

---

### [CRIANDO O REPO NO GITHUB] - 2 minutos

[SLIDE 6 - Tela github.com/new]

**[PROFESSOR DIZ:]**

> Vamos criar. Acessa github.com/new. Ou clica no "+" no canto superior direito.
>
> **Nome do repo:** Usa kebab-case. Nada de espaços ou caracteres especiais. Tipo `task-manager-api` ou `portfolio-site`.
>
> **Descrição:** Uma frase clara. "API REST para gerenciar tarefas" é melhor que "meu projeto".
>
> **[SLIDE 7 - Opções de criação]**
>
> **Visibilidade:** Public pra portfolio. Private pra estudos ou projetos confidenciais.
>
> Agora a parte importante:
>
> - **Add a README file** - Marca. Sempre.
> - **Add .gitignore** - Escolhe o template da sua linguagem. Node, Python, Java, o que você usa.
> - **Choose a license** - Escolhe MIT.
>
> Clica "Create repository".
>
> Pronto. Seu repo já nasce estruturado.

---

### [CLONANDO VIA SSH] - 1 minuto 30 segundos

[SLIDE 8 - Botão Code do GitHub]

**[PROFESSOR DIZ:]**

> Repo criado. Agora vamos trazer pro seu computador.
>
> No GitHub, clica no botão verde "Code". Certifica que tá na aba SSH, não HTTPS.
>
> Copia aquele endereço que começa com `git@github.com:`.
>
> [SLIDE 9 - Terminal]
>
> No terminal:
>
> ```bash
> git clone git@github.com:seu-username/seu-repo.git
> ```
>
> Substitui pelo endereço que você copiou.
>
> Entra na pasta:
>
> ```bash
> cd seu-repo
> ```
>
> Verifica os arquivos:
>
> ```bash
> ls
> ```
>
> Você deve ver: README.md, .gitignore, LICENSE.
>
> Se apareceu os 3, você tá pronto.

---

### [VERIFICANDO CONEXÃO] - 30 segundos

[SLIDE 10 - git remote]

**[PROFESSOR DIZ:]**

> Pra confirmar que a conexão com o GitHub tá certa:
>
> ```bash
> git remote -v
> ```
>
> Você deve ver algo assim:
>
> ```
> origin  git@github.com:seu-username/seu-repo.git (fetch)
> origin  git@github.com:seu-username/seu-repo.git (push)
> ```
>
> Se aparecer git@github.com, tá usando SSH. Perfeito.
>
> Se aparecer https://, você clonou errado. Deleta a pasta e clona de novo pela aba SSH.

---

### [SOBRE O README] - 1 minuto

[SLIDE 11 - Estrutura de README]

**[PROFESSOR DIZ:]**

> Deixa eu te mostrar a estrutura de um README profissional.
>
> No mínimo, você precisa de:
>
> - **Título** - Nome do projeto
> - **Descrição** - O que faz em 1-2 parágrafos
> - **Tecnologias** - Lista do que usou
> - **Instalação** - Como rodar localmente
> - **Uso** - Exemplos práticos
> - **Licença** - Referência ao arquivo LICENSE
>
> [Pausa]
>
> Não precisa ser longo. Precisa ser claro.
>
> Se alguém abrir seu repo e em 30 segundos entender o que é e como rodar, você fez certo.
